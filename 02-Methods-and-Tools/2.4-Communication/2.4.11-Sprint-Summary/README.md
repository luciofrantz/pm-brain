# Sprint Summary Framework

## Introduction

Use this framework and template to send a lightweight async sprint transition brief at the end of every sprint. It covers what the team focused on, what was delivered, what the new sprint is targeting, and what outcomes you expect by the end.

This isn't a sprint review doc — it's a communication artifact. The audience gets the headline; the sprint board screenshot is the detail layer.

**When to use:**
- Sprint closes and a new one starts
- You need to keep delivery leads and senior stakeholders informed without scheduling a meeting
- You want the team to see how their work is being communicated externally — so they can flag if the framing is off

**When this might not be needed:**
- Very small or co-located team with high daily visibility already
- Sprint was a direct continuation with no meaningful topic shift (rare)

---

## Before Using This Framework

⚠️ **Don't jump straight to filling the template.**

Before writing, spend 2 minutes on:
- What actually moved the needle this sprint — separate the 2–3 things that mattered from everything else that got done
- Whether anything slipped that stakeholders should know about (name it before deciding whether to include it)
- What the real goal of the new sprint is — and why it's the right focus *right now*

This is communication, not a status report. The difference shows up in whether stakeholders understand what you're building and why.

See `1-sprint-summary-framework.md` → Step 0 for the full prompt list.

## How to Use This Framework

1. **Review framework:** Read `1-sprint-summary-framework.md` for the principles and structure rationale
2. **Use template:** Open `2-sprint-summary-template.md` and fill in sprint goal, deliverables, and next sprint focus
3. **Paste board screenshots:** Include the actual Dira/Jira/ADO board state — before and after
4. **Send within 24h:** Ideally at sprint close or the morning a new sprint starts
5. **Let the team see it:** Share with team members too — they should know what's being communicated on their behalf

---

## How to Maintain

- **Per sprint:** Fill in and send the template; no archiving needed unless you want a record
- **Quarterly:** Review whether the format is landing well with stakeholders; adjust tone or structure if feedback suggests it
- **If format drifts:** Return to the framework file to recalibrate
